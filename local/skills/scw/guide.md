# 스킬 생성/개선 시 따르는 규칙

skill-creator를 사용하되, 이 프로젝트에서는 추가로 아래를 따른다. Claude Code에서는 플러그인/스킬로, Codex에서는 시스템 스킬로 제공될 수 있다.

Codex 시스템 skill-creator는 지침 로드와 `init_skill.py` 기반 초기화에는 사용할 수 있다. 검증 스크립트는 PyYAML이 필요하므로, 실행 전 같은 Python에서 `import yaml` 가능 여부를 확인한다. 없으면 전역 Python을 변경하지 말고 `%TEMP%` 하위 target에 `python -m pip install --target <target> PyYAML`로 설치한 뒤, 해당 명령 세션에서 `PYTHONPATH=<target>`를 지정해 `quick_validate.py`와 `generate_openai_yaml.py`를 실행한다. `init_skill.py`가 만든 기본 `description: [TODO: ...]`는 YAML 리스트로 해석되어 validator가 실패하므로, 실제 문자열 description으로 채운 뒤 검증한다.

## 사전 학습

스킬을 만들기 전에 아래 블로그를 읽어라. how 최소화, gotchas 축적 등 이 가이드의 원칙이 여기서 나왔다:

## skill-discovery

신규 스킬을 만들기 전에, 기존 스킬 중 유사한 것을 탐색하여 구조/패턴을 참고한다.

- `deploy/skills/` 하위 SKILL.md들의 description을 읽어 유사 스킬 탐색
- 유사 스킬이 있으면 구조를 참고하여 신규 스킬 설계

## 설계 토론 (Capture Intent 보강)

skill-creator의 Capture Intent / Interview 단계에서, 사용자에게 바로 질문하지 않는다. 에이전트팀을 만들어 설계 토론을 거친 뒤 정제된 제안을 가져온다.

### 팀 구성

| 역할 | 책임 |
|------|------|
| 설계자 | 사용자 목표(마이그레이션 문서 등)를 달성하는 설계안을 만든다. 목표 수호 책임 |
| 토론자 | 설계안의 결함을 찾고 도전한다. 구현 후에는 리뷰어 역할도 겸한다 |
| 구현자 | 합의된 설계안을 실행한다. 설계자와 세트로 움직인다 |

설계자+구현자가 한 세트, 토론자가 독립이다.

### 인프라

[CRITICAL] [team-agent](../../../deploy/contexts/team-agent.md)의 규칙을 따른다.

설계 토론 팀은 현재 런타임의 team-agent 경로로 구성한다. Claude Code에서 Agent Teams를 사용할 수 있으면 `team_name`을 지정한다. Codex에서는 허용된 subagent 도구를 사용하되, 같은 역할 작업자를 재사용해 다라운드 맥락을 유지한다. 팀/서브에이전트 구성이 불가능하면 메인이 역할을 직접 수행하고, 그 이유를 사용자 제안에 포함한다.

### 절차

1. **사용자 의도 최소 확인** — 뭘 하고 싶은지, 왜 하는지만 파악한다
2. **설계 토론** — 설계자가 기존 파일을 분석하고 설계안을 만든다. 토론자가 도전한다. 2-3라운드 토론 후 합의. 폴더/파일 구조가 관련된 작업이면, 구조의 합리성(파일 위치, 공용 vs 개별, 참조 경로)도 토론 범위에 포함한다.
3. **사용자에게 제안** — 합의된 설계안을 가져온다. 사용자는 승인/수정/반려한다
4. **구현** — 구현자가 실행한다. 토론자가 리뷰한다

### 왜 이 구조인가

- **설계자가 목표를 수호한다**: 토론자 2명이면 누구도 사용자 목표에 책임이 없다. 설계자가 사용자 목표 달성을 책임지면서 토론자의 도전을 받는다
- **설계 토론이 구현 전에 일어난다**: 구현 후 리뷰에서 설계 문제가 터지면 수정 비용이 높다. 설계안 단계에서 토론하면 비용이 낮다
- **토론자가 설계 맥락을 안다**: 설계 토론에 참여했으므로 구현 리뷰 품질이 높다

## 목적 기반 수단 검증

스킬을 만들거나 기능·규칙을 추가할 때, 그 수단이 스킬 `## 목적`을 달성하는 가장 효과적인 방법인지 먼저 검증한다. 떠오른 방법을 곧바로 박지 않는다 — 직관으로 채택해 더 나은 대안을 놓치는 것을 막기 위함이다.

- 검증은 skill-creator 절차 또는 [설계 토론](#설계-토론-capture-intent-보강) 팀으로 대안 수단과 비교해 수행한다.
- 더 효과적인 수단이 있으면 그것을 역제안한다.
- 프롬프트로 박기 전 더 강한 강제 메커니즘(정적분석·hook) 우선 검토는 이 레포 CLAUDE.md 「프롬프트 작성 원칙」을 따른다.

## how 최소화 원칙

- 새 스킬이든 기존 스킬 개선이든, what(무엇을 하라)만 남기고 how(어떻게 하라)는 최소화한다

how 제거는 **사용자 즉석 리뷰 → 비교 테스트** 두 구간으로 나눈다. AI가 박은 노이즈 how와 사용자가 의도해서 박은 how가 섞여 있어, 제거 후보를 전부 벤치에 넣으면 비용이 과하다. 노이즈는 벤치 없이 떼고, 가치가 의심되는 것만 벤치로 보낸다.

1. **사용자 즉석 리뷰 (벤치 없이)** — 제거 후보 how를 일괄로 떼어낸 모습을 사용자에게 보여주고 즉석 판정을 받는다. AI가 노이즈 여부를 스스로 정하지 않는다 — 후보를 전부 떼서 보여주기만 하고, 노이즈 판정은 사용자가 한다. 사용자가 노이즈(AI가 박은 군더더기)로 판정한 how는 벤치 없이 제거하고 종료한다. 정말 필요한지 사용자도 확신 못 하는 how만 다음으로 넘긴다.
2. **비교 테스트 (벤치)** — 1에서 넘어온 의심 how만 OLD(현재) vs NEW(how 제거)로 검증한다. 판단 기준은 "이걸 빼면 AI가 못 하는가?"이며, AI 자체 판단만으로 유지·삭제를 결정하지 않는다.

## 비교 테스트는 AI가 수행

- 테스트 대상(엉망인 코드 등)은 AI가 만든다
- 테스트 실행 후 비교 분석도 AI가 한다
- 사용자는 AI의 분석 결과를 보고 최종 판단만 한다
- 테스트 버전은 넉넉하게 만든다. OLD/NEW 2개에 한정하지 말고, 변형을 여러 개 만들어서 동시에 비교해도 된다 (예: OLD/NEW/NEW2/NEW3). 동시에 20개 미만으로
- 테스트 결과 분석 후, "제거한 항목이 실제로 필요한 상황을 테스트했는가?" 되돌아본다. 테스트 케이스가 제거 항목을 스트레스 테스트하지 못했다면 추가 테스트를 돌린다
- "테스트 필요 없어 보인다"고 AI가 스스로 판단해 벤치를 스킵하지 않는다 — 벤치 면제는 사용자 즉석 리뷰로만 결정된다. 벤치로 넘어온 how는 어떤 것이든 테스트할 거리를 찾아내서 검증한다

## assertion 사전 검증

eval 실행 전, writer/critic 팀으로 assertions를 검증한다.

### 팀 구성

| 역할 | 책임 |
|------|------|
| writer | assertion 초안 작성 |
| critic | 각 assertion에 아래 3가지로 도전. 통과하지 못하면 수정 또는 DROP 요구 |

### critic 검증 기준

1. **컨텍스트 충족**: 이 assertion을 pass하려면 에이전트가 무엇을 읽어야 하는가? eval setup에 포함됐는가?
2. **스킬 규칙 매핑**: 스킬의 어떤 구체적 규칙이 이 assertion의 pass를 유도하는가? 매핑 불가면 제거.
3. **차별성**: 스킬 없는 baseline sonnet이 이걸 pass할 수 있는가? Yes면 강화하거나 교체.

합의될 때까지 라운드를 반복하고, 최종 assertions만 evals.json에 기록한다.

## 자기 검증

- 리뷰 완료 후, 자신의 리뷰를 비판적으로 되돌아본다. 테스트 케이스가 충분했는지, 빠진 엣지 케이스는 없는지 스스로 따져본다. 부족한 테스트 케이스를 발견하면 사용자 승인 없이 바로 추가 벤치마크를 돌린다

## 텍스트 작성 자가검토

SKILL.md·specialized·guide 등 프롬프트 텍스트를 작성·수정한 직후 자가검토한다. 외부 리뷰(「프롬프트 감사」)와 별개로, 작성자가 즉시 잡을 수 있는 패턴을 놓치지 않기 위함이다.

### 목적 섹션 확인

SKILL.md를 새로 만들거나 수정했으면 H1 바로 아래에 `## 목적`이 있는지 확인한다. 없으면 추가한다 (「산출물 규칙」).

### 숫자 하드코딩 잡기

작성·수정 후 grep으로 본문에 박힌 구체적 개수를 검사한다. 항목 개수가 본문에 박히면 항목이 늘거나 줄 때 본문도 같이 손봐야 하는 결합이 생긴다.

- 검사 패턴: `\d+(개|문항|단계|건|항목)` 등
- 일반화 표현으로 교체 (`5문항` → `자가검증 문항`, `3개 단계` → `각 단계`)
- 글로벌 룰 「구체적인 개수를 본문에 하드코딩하지 않는다」(이 레포 CLAUDE.md)의 자동화 절차

### 사용자 발화 일반화

사용자가 직접 한 발화(특히 1인칭·구어·맥락 의존 표현)를 SKILL.md에 그대로 옮기지 않는다. SKILL.md는 공개 문서이고 다른 세션의 AI가 읽는 글이라, 작성 시점의 사용자 개인 사정·구어가 그대로 들어가면 의미가 깨진다.

- 예: "난 자세히 안 봐" → "사용자가 본문을 매번 검수하지 않을 수 있다는 전제에서"
- 예: "그냥 잘 재현만 되게" → "재현 가능성이 본 절차의 목표"
- 일반화 후 박는다. 사용자 개인 발화를 그대로 옮기는 것은 금지

### 강조 라벨 절제

"항상 X 하라"를 문서에 반영할 때:
- 조건부 게이트(`(선택)`, `필요하면`, `필요시`)를 제거해 평범한 단언 절차문으로 만든다
- `(필수)`, `항상`, `반드시`, `묻지 않는다` 같은 강조 라벨은 덧붙이지 않는다

## 산출물 양식 원칙

- 산출물 예시(마크다운 코드블록)로 양식을 고정하지 않는다
- 대신 **필수 필드만 명시**하고, 구체적 마크다운 형태는 AI 자유
- 이유: 예시로 고정하면 더 나은 구조를 만들 기회를 막는다. 필수 필드만 지정하면 양식 일관성과 유연성을 둘 다 잡을 수 있다
- 예시를 제거할 때, 그 예시가 유도하던 필수 필드/행동이 있는지 확인한다. 있으면 예시는 제거하되 필수 필드명은 남긴다

## 산출물 규칙

- SKILL.md는 H1 제목 바로 아래 첫 섹션으로 `## 목적`을 두고, 스킬의 목표를 한 줄로 적는다. frontmatter `description`은 트리거(언제 부르는가) 판정용이라 매 세션 로드되고, `## 목적`은 호출된 AI가 "왜 이걸 하는가"를 본문 진입에서 잡는 용도다 — 역할이 다르므로 둘 다 둔다. 목적이 이미 lede 한 줄로 적혀 있으면 그 줄을 `## 목적` 섹션으로 승격하고 중복 생성하지 않는다. 「왜 이 스킬이 필요한가」(존재 근거)는 목적과 별개 절이다.
- 스킬은 `deploy/skills/{skill-name}/`에 생성
- 드래프트 상태면 `STATUS.md`를 함께 생성하여 진행 상황과 남은 작업 기록
- 실사용 후 안정되면 `STATUS.md` 삭제
- 커밋 메시지 scope에 스킬명을 쓸 수 있도록 git commit-msg 훅(`scripts/commit_msg.py`)의 `SCOPE_ENUM`에 추가

## gotchas 축적

- 스킬을 처음 만들 때 gotchas는 비워둔다
- 실사용 중 AI가 반복 실패하는 패턴을 발견하면 SKILL.md에 gotchas 섹션으로 추가

## 서브에이전트 위임 구조 테스트

스킬 내부에서 추가 에이전트를 띄우는 패턴은 skill-creator 표준 방식(서브에이전트에서 스킬 실행)으로 테스트할 수 없다. 메인 에이전트가 직접 오케스트레이션한다. 상세는 [eval-delegation.md](eval-delegation.md) 참조.

## 벤치 정리

벤치 완료 후 **벤치 안 한 상태**로 복귀한다. 산출물·환경 변경이 사용자 작업 디렉토리에 남으면 다음 작업을 방해하고, 사용자가 직접 청소하게 된다. 빠짐없이 다음을 정리한다.

- **workspace 폴더**: `rm -rf <bench-dir>` (테스트 문서, eval 결과, 스냅샷, 입출력 사본, 임시 산출물 모두)
- **임시 브랜치·워크트리**: 벤치용으로 만든 브랜치는 삭제, 워크트리는 `git worktree remove`. 사용자가 작업 중이던 브랜치로 복귀시킨다
- **gitignore/exclude 추가분**: `.git/info/exclude`, `.gitignore`에 벤치용으로 추가한 줄 제거
- **임시 환경 변경**: 환경변수, 임시 설치된 의존성, 수정된 설정 파일 등 원복

정리 후 `git status`와 작업 디렉토리를 직접 확인해 흔적이 0이 됐는지 검증한 뒤 작업을 종료한다. 일부 정리만 하고 종료하지 않는다.

## gotchas

- **벤치마크 인프라 실패 시 스킵 금지**: OLD 스냅샷 경로 오류 등으로 비교 대상이 무효화되면, 원인을 수정하고 해당 run만 재실행한다. "additive 변경이라 괜찮다"는 합리화로 넘어가지 않는다.
