# 서브에이전트 위임 구조 Eval 가이드

skill-creator의 표준 eval(서브에이전트에서 스킬 실행)은 **스킬 내부에서 추가 에이전트를 띄우는 패턴**을 테스트할 수 없다. 하위 에이전트는 다시 팀/서브에이전트를 구성할 권한이 없거나 제한될 수 있기 때문이다.

## 혼합 스킬 분해 — 일반 원칙

오케스트레이션과 위임이 섞인 스킬(workflow처럼 세션이 또 서브에이전트를 띄우는 구조)은 한 단위로 벤치하지 못한다. 아래 4원칙으로 분해한다.

### 1. 깊이 제약 → 분해

서브에이전트는 다시 서브에이전트를 못 만든다. 벤치 단위를 spawn 깊이로 가른다.

- **내부에서 또 spawn하는 오케스트레이터** → 메인이 직접 구동(depth-1 위임은 가능). 아래 「메인 에이전트가 직접 오케스트레이션」.
- **더 안쪽의 단일 에이전트 말단**(추가 spawn 없음) → 서브에이전트로 **병렬 fan-out**. 메인 직접에서 잃는 병렬·fresh-eyes를 여기서 회복한다.

세션 단위로 묶지 말 것 — 세션이 내부 spawn을 하면 그 세션은 오케스트레이터다.

### 2. 벤치 단위 = 공유되는 절차, 인스턴스 아님

같은 절차 정의를 여러 곳에서 반복 적용하는 스킬은 그 **절차가 벤치 단위**다. 절차를 고치면 "그 절차를 여러 입력에 대해" 재벤치하지, 인스턴스 체인을 처음부터 다시 돌리지 않는다.

### 3. fixture = 즉석 mock 기본, 체이닝 검증만 harvest

말단의 입력(선행 산출물·코드·중간 상태)은 그 자리에서 규칙(계약)에 맞게 만든다(mock). 저장한 fixture가 없으니 스킬을 고쳐도 stale될 게 없다.

예외 — **경계 체이닝 검증**(말단이 직전 말단의 *실제* 출력을 제대로 먹는가)은 mock으로 못 잡는다. mock은 곧 "내가 가정한 계약"이라 가정이 틀리면 불일치가 숨는다(자기증명 루프). 이때만 실제 직전 말단을 돌린 출력을 harvest한다.

mock 작성·분기 트리거 주의는 아래 gotchas 「선행 산출물 체크」·「mock 데이터는 모든 분기 트리거」 참조.

### 4. 순서 강제 아님

mock 덕에 선행 단계를 안 거치고 아무 말단이나 독립으로 벤치할 수 있다. "선행부터 순서대로"는 체이닝 검증(원칙 3 예외)일 때만 필요한 fallback이다.

### 채점 — 오라클 독립성

각 말단 출력의 채점 기준은 **그 말단이 따라야 할 1차 소스 직접 대조**다(요구사항 원본·figma·컨벤션 1차 소스 등). 고정 rubric은 두지 않고 AI가 그때그때 판단한다. 단 **스킬 자신의 내부 reviewer로 채점하지 않는다** — 벤치 대상이 채점자가 되면 자기증명 루프다.

## 해결: 메인 에이전트가 직접 오케스트레이션

원칙 1에서 "메인 직접"으로 분류된 오케스트레이터 부분을 돌리는 방법이다. 최상위 세션의 메인 에이전트가 스킬의 지시를 따라 직접 실행한다.

| | skill-creator 표준 | 메인 직접 실행 |
|---|---|---|
| 실행 주체 | 서브에이전트 | 최상위 세션 |
| 에이전트 도구 | 없음 또는 제한됨 | 현재 런타임에서 제공되는 범위 |
| 병렬 A/B | 가능 | 불가 (순차) |
| 적합한 대상 | 단일 에이전트로 완결되는 스킬 | 내부에서 서브에이전트를 띄우는 스킬 |

## 절차

1. 가상 프로젝트 준비 (git init, 최소한의 파일, 스킬이 요구하는 산출물)
2. 스킬 파일을 읽고 지시를 그대로 따름
3. 스킬이 서브에이전트 위임을 지시하면 team-agent 규칙에 맞는 실제 경로로 실행
4. 각 단계에서 검증 항목 확인
5. 서브에이전트의 토큰/시간을 task notification에서 수집

## 유효 범위

**검증 가능**: 위임 트리거 인식, 오케스트레이션 사이클, 서브에이전트 간 산출물 전달, 커밋 순서

**검증 불가**: before/after 정량 비교 (병렬 A/B 불가), 처음 보는 AI의 반응 (메인이 스킬 내용을 이미 앎)

## gotchas

- **방향 확인**: "출력이 맞는가"가 아니라 "분기·순서·에이전트 구성이 맞는가"를 검증한다. 테스트 입력은 각 분기 경로를 타도록 설계한다.
- **시나리오 격리**: 독립된 테스트 시나리오는 별도 세션에서 실행한다. 한 세션에서 같은 팀/작업자를 여러 시나리오에 걸쳐 재사용하면 컨텍스트가 오염된다.
- **선행 산출물 체크**: 테스트 대상이 이전 step의 산출물에 의존하면, mock을 만들어야 한다. 테스트 대상 스킬만 보면 이 의존성을 놓친다.
- **mock 데이터는 스킬의 모든 조건 분기를 트리거해야 한다.** "오케스트레이션만 보면 된다"고 클린 데이터만 넣으면 차단/에러 경로가 발동하지 않는다. 게이트, 필터링, 루프 종료 등 분기가 있으면 각 경로를 트리거하는 입력을 포함한다.
