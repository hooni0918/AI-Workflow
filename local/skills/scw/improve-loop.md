# 의심 지점 개선 루프

스킬·프롬프트의 의심 지점을 AI가 분류·검증·적용한다. 사용자 판정을 받지 않고 커밋까지 자동 수행한다 ("프롬프트 감사" 모드와 구분).

## 트리거

다음 의도 중 하나라도 잡히면 이 모드 진입:
- "의심 지점 찾아 개선"
- "벤치 돌려서 다듬어"
- "비판적으로 따져"
- "끝까지 다 개선"
- 명시 키워드 없이도 사용자가 "스킬 보고 개선점 찾아줘" 의도면 적용

## 사전 작업 (워크트리)

작업 전 git 상태 점검:
- 현재 브랜치가 master/main이 아니거나 uncommitted가 있으면 워크트리 사용
- master에서 워크트리 + 브랜치 생성. 이름은 작업 대상 기반 (예: `feature/<target>-improve`)
- 사용자에게 워크트리 경로·브랜치명 보고

작업 종료 시 워크트리 삭제는 사용자 지시를 받는다. 자동 삭제하지 않는다.

## 자동 진행 디폴트

라운드 사이 사용자 확인 질문 금지. 한 라운드(스캔→분류→적용→보고)가 끝나면 잔류 의심 지점이 있는 한 자동으로 다음 라운드에 진입한다. "다음 라운드 진행할까요?", "다음은 어느 파일 볼까요?" 같은 진행 의사 확인을 사용자에게 묻지 않는다.

한 의심 지점을 처리한 뒤 다음 의심 지점으로 넘어가는 것도 라운드 진행에 포함된다. 매번 의사 확인을 받지 않는다.

사용자가 명시적으로 "멈춰", "여기까지", "다음 세션으로" 라고 말할 때만 중단. 결과 보고는 매 라운드 짧게(채택/반려·근거 한 줄) 흘려두고, 사용자가 끼어들면 그때 방향 전환.

## 5단계 워크플로우

### 1. 스캔

대상 스킬과 참조 파일을 모두 Read한다. 의심 지점을 모두 수집한 뒤 분류로 넘어간다 (한 건씩 보고 처리하지 않는다).

수집 관점: 사문, 중복, 정합성 모순, how 과잉(guide.md 「how 최소화 원칙」), 가독성, 정의 모순, 사용되지 않는 분기, 모호한 표현 등.

### 2. 2-tier 분류

| Tier | 기준 | 검증 요구 |
|------|------|-----------|
| 객관적 정리 | 사문(어디서도 참조 안 됨)·중복(다른 곳에 그대로 있음)·모순·버그 — **외부 사실로 증명 가능**한 것만 | 불필요 — 즉시 수정 |
| 벤치 필수 | 위에 해당하지 않는 모든 의심 지점. "이 줄을 빼도 같은 동작" 같은 행동 예측은 전부 벤치로 보낸다 | guide.md "비교 테스트" 절차로 OLD vs NEW 검증. 대상이 `deploy/rules/` 등 지침/규칙 나열 문서면 추가로 [specialized/rule-ablation-bench.md](specialized/rule-ablation-bench.md) 적용 |

판단 원칙: 의심되면 벤치 필수 쪽으로 분류한다. "이건 명백해 보인다"고 객관적 정리로 미루지 않는다. 분류 근거가 약하면 벤치로 보낸다.

### 3. 적용

- 객관적 정리: 수정 → 항목당 커밋 1개
- 벤치 필수: guide.md 비교 테스트 → 결과 분석 → 채택/반려 결정 → 채택 시 커밋
- 스코프 외 발견: 5단계로 백로그 이관

수정한 파일만 명시 staging (`git add -A` 금지, CLAUDE.md 규칙).

### 4. 보고

사용자에게 표 형식으로 보고. 필수 컬럼:
- 커밋 해시
- 무엇을 수정했는가
- 왜 수정했는가
- 어떤 벤치 결과가 근거인가 (없으면 분류 명시: 객관적)

추상 설명 금지. 벤치 결과는 OLD/NEW 실제 출력의 핵심 차이를 1-2줄로 인용한다.

### 5. 스코프 외 이슈 보고

이 레포에는 별도 백로그 트래커가 없다. 스코프 외 이슈는 파일로 적재하지 않고, 라운드 보고에 **인수인계 가능한 분량으로** 적어 사용자에게 넘긴다 (짧게 요약하지 않는다). 사용자가 별도 추적을 원하면 그때 위치를 정한다.

필수 항목:
- 발견 맥락 (어느 세션 어느 단계)
- 현재 구조
- 문제 상세
- 왜 문제인가
- 재현 시나리오
- 해결 방향 후보
- 검증 방법
- 관련 파일 / 관련 커밋

## 자기 검증과의 연결

보고 후 guide.md 「자기 검증」 적용. 빠진 테스트 케이스를 발견하면 사용자 승인 없이 추가 벤치를 돌린다.

## gotchas

- 검증 없이 수정한 항목을 "벤치 기반"이라고 보고하지 말 것 — tier를 정확히 명시한다 (사용자가 이 패턴을 3회 이상 추궁한 적 있음)
- "의심된다 → 검증한다"가 원칙. "이건 명백해 보인다"고 벤치 스킵 금지 (guide.md 「비교 테스트는 AI가 수행」 재확인)
- 100줄 이상 변경이 예상되면 plan mode 진입 (CLAUDE.md 전역 규칙)
- workspace 폴더는 작업 종료 전 반드시 삭제 (guide.md 「벤치 정리」)
- 워크트리 자동 삭제 금지 — 사용자 지시 받기
- 백로그 이관 시 항목을 짧게 요약하면 인수인계 불가. tier-{N}/{target}.md 형식의 다른 기존 항목을 참고해 분량을 맞춘다
