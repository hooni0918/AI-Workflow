# 아키텍처 경계 원칙 (마스터)

아키텍처를 가리지 않고 통하는 경계 원칙을 다룬다 — 무엇이 무엇에 의존해야 하는가, 변경을 어떻게 격리하는가, 외부와의 접점을 어디서 끊는가. 특정 아키텍처(Clean Architecture·DDD·MVVM·VIPER 등)·레이어 목록·빌딩블록·네이밍은 고르지 않는다. 그것들은 이 원칙을 구현하는 *한 방식*일 뿐이며, 구체는 프로젝트가 공급한다.

**구체의 출처.** 정확한 레이어 목록·의존 허용표·빌딩블록·경계 규칙은 대상 프로젝트가 1차 출처다 — `.claude/docs/architecture.md`(아키텍처 형태), `layer-rules.json`(기계 대조 가능한 import·의존 규칙), `.claude/docs/project-profile.md`(UI 프레임워크·언어 버전 등 프로필). 이 파일들이 있으면 그것을 따른다. 없으면 임의로 가정하지 말고 사용자에게 묻고, 묻지 못하는 상황이면 **가정 라벨**(예: "가정: 도메인↔인프라 2층, 단방향")을 명시한 뒤 진행한다. 이 문서는 그 규칙들이 "왜 그렇게 생겼는가"의 근거만 담는다.

## 왜 구체가 아니라 추상에 의존하는가 (DIP)

상위(정책·규칙) 코드가 하위(구체 구현)를 직접 알면, 상위 로직이 그 구현의 변경에 묶인다. 그래서 의존 방향을 뒤집는다 — 상위는 **추상**(`protocol`/인터페이스)만 정의하고, 하위가 그 추상을 구현한다. 어디서 무엇을 끼워 넣을지(컴포지션 루트)는 런타임에 구현체를 주입한다.

- **상위가 테스트 가능해진다**: 추상에만 의존하면, 테스트에서 가짜 구현을 주입해 외부 의존(네트워크·DB·시계 등) 없이 검증할 수 있다.
- **구현을 갈아끼울 수 있다**: 원격을 로컬·목으로 바꿔도 상위는 그대로다. 의존이 추상에 걸려 있기 때문이다.
- **명시적 주입이 이를 강제한다**: 구현체를 내부에서 직접 생성하지 않고 밖에서 받으므로, 무엇에 의존하는지가 타입 시그니처에 드러난다. 숨은 결합이 생기지 않는다.

> 이 의존 역전은 코드만 보면 평범한 추상/주입으로 보인다. 리뷰·인계에서 "여기서 구체 타입을 직접 참조했다"를 지적할 때는 위배된 규칙(DIP)을 함께 명시한다. 어느 방향이 상위이고 어느 추상으로 끊어야 하는지는 프로젝트 `.claude/docs`·`layer-rules.json`을 출처로 본다.

## 왜 단방향인가 — 변경의 격리

레이어 의존을 한 방향으로만 두는 이유는 **변경의 파급 범위를 미리 묶어두기 위함**이다. 하위가 상위를 모르면, 상위를 고칠 때 하위는 다시 컴파일·검토할 필요가 없다. 변경의 영향이 의존 화살표를 거슬러 올라가지 못한다.

- **역방향이 한 번 뚫리면** 격리가 무너진다. 하위가 상위를 알기 시작하면, 어느 변경이 어디까지 번지는지 추적 불가능해지고 결국 전체가 한 덩어리로 굳는다.
- **수평(같은 레이어 모듈끼리) 결합도 같은 문제**다. 한 모듈이 동급 다른 모듈을 직접 알면 둘을 따로 떼어낼 수 없게 묶인다. 공유가 필요하면 공통부를 더 아래(공용) 레이어로 내린다.
- 그래서 "어디 둘지 애매한" 코드를 가까운 레이어에 임시로 끼워 넣지 않는다. 한 번 잘못 걸린 의존이 단방향성을 깨고, 그 순간 위 두 보장이 사라진다.

> 어떤 레이어가 무엇을 import해도 되는지(허용표)는 프로젝트의 `layer-rules.json`이 1차 출처다. 레이어 수·이름은 레포마다 다르므로 이 문서는 정하지 않는다.

## 왜 경계에 번역막을 두는가 (Anti-Corruption)

외부(네트워크·외부 API·서드파티 SDK·영속화 스키마)의 형태와 오류는 내부 사정과 무관하게 바뀐다. 이걸 내부 정책이 직접 받으면, 외부가 바뀔 때마다 내부 로직이 따라 흔들린다. 그래서 경계 레이어가 **번역막** 역할을 한다 — 외부 오류를 내부 오류로, 외부 타입을 내부 타입으로 옮긴 뒤에야 위로 올린다.

- **오류 번역**: 외부 오류는 경계에서 내부 오류로 번역된다. 내부 정책은 외부 오류 타입(예: 통신·디코딩 오류)을 보지 못한다. "통신이 어떻게 실패하는가"가 아니라 "내 도메인 관점에서 무엇이 잘못됐는가"로만 분기하게 하기 위함이다.
- **스키마 번역**: 외부가 주는 형태를 내부 모델로 옮긴다. 내부 모델이 외부 스키마를 직접 떠안으면 그 외부 계약에 다시 묶인다.
- **이 막이 없으면** 외부 사정이 내부 분기문에까지 새어들어, 외부 계층을 바꿀 때 내부 코드를 같이 고쳐야 한다. 막이 결합을 한 곳(경계)에 가둔다.

## 추상↔구체의 분리를 경계로 본다

각 레이어가 "정책(무엇을·왜)"을 다루는지 "메커니즘(어떻게)"을 다루는지로 책임을 가른다 — 둘을 한 타입에 합치면 정책이 메커니즘의 변경에 묶인다.

- **정책 쪽**: 도메인 규칙·결정·경계 추상의 *정의*. 외부 관심사(직렬화 형식·표시 문자열·전송 프로토콜)는 담지 않는다 — 담으면 정책이 특정 메커니즘에 묶인다.
- **메커니즘 쪽**: 경계 추상의 *구현*, 외부 호출, 타입·오류 번역. 정책 규칙을 여기에 두지 않는다.
- 식별자·값을 원시 타입 대신 의미 있는 타입으로 감싸는 것도 같은 맥락이다 — 서로 다른 개념이 타입으로 구분되어, 잘못 섞이는 결합을 컴파일 시점에 막는다.

경계를 넘는 책임을 "편의상" 가까운 레이어에 끼워 넣지 않는다. 한 번 잘못 들어간 책임이 위의 단방향·번역막·추상/구체 분리를 동시에 무너뜨린다. 무엇이 어느 쪽에 속하는지의 구체 분류는 프로젝트 `.claude/docs/architecture.md`를 출처로 본다.

## 이 원칙을 구현하는 한 방식: Clean Architecture / DDD

위 네 원칙(DIP·단방향·Anti-Corruption·추상/구체 분리)은 아키텍처 이름과 무관한 골격이다. Clean Architecture + DDD는 이를 구현하는 *하나의 예시*다 — 도메인을 가장 안쪽에 두고, 빌딩블록(예: Entity·UseCase·Repository 추상)으로 정책을 표현하며, 바깥 레이어가 그 추상을 구현하고 외부를 번역한다. MVVM·VIPER·헥사고날 등도 같은 골격을 다른 어휘로 구현할 수 있다.

마스터는 그중 무엇도 고르지 않는다. 어떤 아키텍처를 쓰는지, 레이어가 몇 층이고 무엇으로 부르는지, 어떤 빌딩블록·네이밍을 쓰는지는 **사용 프로젝트가 공급**한다.

- 구체 레이어 분류·빌딩블록 예시 → 마스터 `examples/`(참고용) 또는 마스터 `templates/`(채워 쓰는 틀)
- 그 프로젝트의 확정 정의 → 대상 프로젝트의 `.claude/docs/architecture.md`·`layer-rules.json`·`.claude/docs/project-profile.md`
