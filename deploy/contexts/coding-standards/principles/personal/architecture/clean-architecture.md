# Clean Architecture 원칙

레이어를 단방향으로 가르고 도메인을 인프라에서 떼어내는 이유와 그 경계 판단을 다룬다. 기계 대조 가능한 "~해라/~금지"(어느 레이어가 무엇을 import해도 되는가, 빌딩블록 형태)는 [rules/personal/clean-architecture/layer-rules.md](../../../rules/personal/clean-architecture/layer-rules.md)가 담당한다.

프로젝트 고유의 정확한 레이어 목록·경계·의존표는 대상 프로젝트의 `.claude/docs/clean-architecture.md`가 1차 출처다. 그 파일이 있으면 그것을 따르고, 이 문서는 그 규칙들이 "왜 그렇게 생겼는가"의 근거만 담는다.

## 왜 구체가 아니라 추상에 의존하는가 (DIP)

`UseCase`가 특정 `Repository` 구현(네트워크로 가져오는 `...Impl`)을 직접 알면, 도메인 로직이 그 구현의 변경에 묶인다. 그래서 의존 방향을 뒤집는다 — 상위(Domain)는 `protocol`만 정의하고, 하위(Data)가 그 `protocol`을 구현한다. 컴포지션 루트(AppDependencies)가 런타임에 구현체를 끼워 넣는다.

- **도메인이 테스트 가능해진다**: `UseCase`가 `protocol`에만 의존하면, 테스트에서 가짜 구현을 주입해 네트워크 없이 검증할 수 있다.
- **구현을 갈아끼울 수 있다**: 원격 저장소를 로컬·목으로 바꿔도 도메인은 그대로다. 의존이 추상에 걸려 있기 때문이다.
- **수동 생성자 주입이 이를 강제한다**: 구현체를 내부에서 `new` 하지 않고 밖에서 받으므로, 무엇에 의존하는지가 타입 시그니처에 드러난다. 숨은 결합이 생기지 않는다.

> 이 의존 역전은 코드만 보면 평범한 `protocol`/`init` 주입으로 보인다. 리뷰·인계에서 "여기서 구체 타입을 직접 참조했다"를 지적할 때는 위배된 규칙(DIP)을 함께 명시한다.

## 왜 단방향인가 — 변경의 격리

레이어 의존을 한 방향(위→아래)으로만 두는 이유는 **변경의 파급 범위를 미리 묶어두기 위함**이다. 하위가 상위를 모르면, 상위를 고칠 때 하위는 다시 컴파일·검토할 필요가 없다. 변경의 영향이 의존 화살표를 거슬러 올라가지 못한다.

- **역방향이 한 번 뚫리면** 격리가 무너진다. 하위가 상위를 알기 시작하면, 어느 변경이 어디까지 번지는지 추적 불가능해지고 결국 전체가 한 덩어리로 굳는다.
- **수평(같은 레이어 모듈끼리) 결합도 같은 문제**다. Feature가 다른 Feature를 직접 알면 두 기능을 따로 떼어낼 수 없게 묶인다. 공유가 필요하면 공통부를 더 아래 레이어로 내린다.
- 그래서 "어디 둘지 애매한" 코드를 가까운 레이어에 임시로 끼워 넣지 않는다. 한 번 잘못 걸린 의존이 단방향성을 깨고, 그 순간 위 두 보장이 사라진다.

## 왜 도메인 앞에 번역막을 두는가 (Anti-Corruption Layer)

인프라(네트워크·디코딩)의 스키마와 오류는 도메인 사정과 무관하게 바뀐다. 이걸 도메인이 직접 받으면, API가 바뀔 때마다 도메인 로직이 따라 흔들린다. 그래서 Data 레이어가 **번역막** 역할을 한다 — 인프라 오류를 도메인 오류로 옮기고, 인프라 타입을 도메인 타입으로 옮긴 뒤에야 위로 올린다.

- **오류 번역**: 인프라 오류는 Data에서 도메인 오류로 번역된다. 도메인은 인프라 오류 타입(`NetworkError`·`DecodingError` 등)을 보지 못한다. 도메인 로직이 "통신이 어떻게 실패하는가"가 아니라 "도메인 관점에서 무엇이 잘못됐는가"로만 분기하게 하기 위함이다.
- **이 막이 없으면** 인프라의 사정이 도메인 분기문에까지 새어들어, 통신 계층을 바꿀 때 도메인 코드를 같이 고쳐야 한다. 막이 결합을 한 곳(Data)에 가둔다.

## 왜 DTO와 Entity를 나누는가 — API 결합 끊기

DTO는 외부 API의 형태를 그대로 받는 타입이고, Entity는 도메인이 다루는 형태다. 둘을 하나로 합치면 도메인 모델이 API 스키마에 직접 묶여, 응답 필드 하나가 바뀌어도 도메인 전체가 영향을 받는다.

- **DTO는 Data 내부에 가둔다**: Domain은 DTO를 보지 못한다. API 형태가 도메인 경계를 넘지 않게 하기 위함이다. 둘 사이 변환도 Data가 맡는다.
- **그래서 Entity가 외부 직렬화 형식을 직접 준수하지 않는다**: Entity가 직렬화 형식을 직접 떠안으면 그 형식(=외부 계약)에 다시 묶인다. 직렬화는 DTO의 일이고, Entity는 도메인 의미만 표현한다.
- 식별자를 원시 타입이 아니라 값 객체로 감싸는 것도 같은 맥락이다 — 서로 다른 도메인 식별자가 타입으로 구분되어, 잘못 섞이는 결합을 컴파일 시점에 막는다.

## 어디까지가 각 레이어의 일인가

- **Domain의 일**: 도메인 규칙·`UseCase`·경계 `protocol` 정의. UI 관심사(포맷팅·화면 상태·표시 문자열)는 담지 않는다 — 담으면 도메인이 특정 UI에 묶인다.
- **Data의 일**: 경계 `protocol` 구현, 인프라 호출, DTO↔Entity 변환, 오류 번역. 도메인 규칙을 여기에 두지 않는다.
- **상위(Feature/App)의 일**: 사용자 입력을 `UseCase` 호출로 잇고 결과를 화면에 반영. 인프라 타입을 직접 만지지 않는다 — 하위 레이어가 번역해 올린 도메인 타입만 다룬다.

경계를 넘는 책임을 "편의상" 가까운 레이어에 끼워 넣지 않는다. 한 번 잘못 들어간 책임이 위의 단방향·번역막·DTO 분리를 동시에 무너뜨린다.
