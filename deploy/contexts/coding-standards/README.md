# 코딩 스탠다드

iOS 코딩 표준을 **정의·강제하는 방법** 모음입니다. 특정 아키텍처·UI 프레임워크·레이어 구성·네이밍 규칙을 고르지 않습니다. 범용 규칙과 원칙, 그리고 프로젝트가 자기 스택을 꽂아 넣는 구조를 제공합니다.

## 정체성

- 이 디렉토리는 **마스터 표준**입니다: iOS 코딩 표준을 어떻게 정의하고 강제할지(범용 규칙·원칙·정적분석 사다리·강제 메커니즘 우선순위)를 담습니다.
- 구체 사실 — 아키텍처(예: Clean Architecture/DDD), UI 프레임워크(예: UIKit/SwiftUI), 레이어 구성·의존 방향, 네이밍 접미사 — 은 **사용 레포가 공급**합니다.
- 마스터 규칙은 "프로젝트에 해당 config가 있으면 그것을 1차 출처로 따르고, 없으면 사용자에게 묻거나 안전 가정(가정 라벨)을 명시한다"는 원칙으로 작성됩니다.

## 다루는 기준

**범용** — 어느 iOS 프로젝트에나 적용되는 기준:

- **Swift 언어 규칙**: 강제 언래핑·force-try 금지, 접근제어 명시 등 정적분석으로 기계 대조 가능한 규칙
- **네이밍 규율**: Swift API Design Guidelines 기반 원칙(접미사·계층별 명명 등 구체값은 프로젝트가 공급)
- **아키텍처 원칙**: 레이어 단방향 의존, 경계·응집도·결합도 같은 판단 기준(구체 레이어·빌딩블록은 프로젝트가 공급)
- **정적분석 사다리**: 코드 컨벤션을 강제 메커니즘이 강한 순서로 내리는 방법

**examples** — 프로젝트별로 대체하는 예시:

- 특정 스택(특정 아키텍처/UI 프레임워크/레이어)을 쓰는 프로젝트의 구체 예시입니다.
- 마스터 표준이 아니므로, 사용 레포는 자기 `.claude/docs`에 자기 것으로 정의해 대체합니다.

## 프로젝트가 공급하는 config

구체 사실은 사용 레포의 다음 파일을 **1차 출처**로 따릅니다:

- `.claude/docs/architecture.md` — 채택한 아키텍처 방법론과 그 레이어 정의
- `layer-rules.json` — 레이어 경계·의존 방향의 기계 대조용 정의
- `.claude/docs/project-profile.md` — 스택 프로필(UI 프레임워크, 모듈 구성, 네이밍 규약 등)

해당 config가 없으면 사용자에게 묻거나, 부득이할 경우 **안전 가정**을 세우고 그 판단에 가정 라벨을 붙여 명시합니다.

## 상황에 맞는 기준 적용

모든 기준을 모든 프로젝트에 일률적으로 적용하지는 않습니다. 범용 규칙·원칙(`rules/`·`principles/`)은 어떤 iOS 프로젝트에든 적용하고, 특정 스택 예시(`examples/`)는 직접 적용하지 않고 프로젝트가 자기 아키텍처·UI·네이밍으로 대체합니다. 컨텍스트에 맞는 수준을 고르는 것도 엔지니어링 판단의 일부입니다.

## 강제 메커니즘 우선순위

규칙을 글로 적기 전에, 더 강한 강제 메커니즘으로 내릴 수 있는지 먼저 봅니다. 코드 컨벤션은 **정적분석(SwiftLint·SwiftFormat·SPM `dependencies` 대조) → hook → 프롬프트 규칙** 순으로 시도합니다. 정적분석으로 잡히는 것은 정적분석에 맡기고, 의미·맥락 판단이 필요한 것만 규칙 문서로 남깁니다.

단, **코드 규칙의 구체값은 프로젝트가 공급**합니다. 정적분석 설정에 어떤 규칙을 켜는지, dependencies 대조에 쓸 레이어가 무엇인지 같은 구체값은 마스터가 고르지 않고 사용 레포의 config(위 「프로젝트가 공급하는 config」)에서 가져옵니다. 마스터는 "정적분석을 우선으로 시도하라"는 사다리만 강제합니다.

인덱스와 로드 규칙은 [map.md](map.md)를 참고하세요.
