# Testing Strategy — Assumptions

iOS 코드의 테스트·CI·검증 전제를 모은 **단일 출처**입니다. 단, 스택 자체(테스트 프레임워크·동시성 모드·Observation 등)는 이 마스터가 단정하지 않는다 — 그 값은 프로젝트 프로필(`.claude/docs/project-profile.md`)·`architecture.md`가 1차 출처다. 없으면 사용자에게 묻거나 안전 가정(가정 라벨)으로 둔다. 아래 항목 중 "가정"으로 표기된 것은 프로젝트가 실측 설정(CI 워크플로·스냅샷 픽스처 등)을 공급하기 전까지의 **가정**이며, 다른 스킬·규칙 문서는 테스트 전제를 본문에 다시 적지 않고 이 문서를 가리킵니다.

## [CRITICAL] 가정 라벨 규칙

- 아래 "가정"으로 표기된 항목은 실측으로 확정되기 전까지 **단정하지 않는다**. 산출물·리뷰 보고에서 인용할 때 "(가정)" 라벨을 유지한다.
- 가정이 실제 설정으로 확정되면, 이 문서에서 해당 항목을 "확정"으로 승격하고 근거(파일 경로·CI 잡 이름 등)를 함께 적는다.

## 빌드·테스트 명령

빌드·테스트의 정답 명령(빌드 오라클, 모듈별 테스트 러너, 시뮬레이터 조회 절차 등)은 이 마스터가 적지 않는다 — 프로젝트 프로필(`.claude/docs/project-profile.md`)의 빌드 오라클·테스트 명령이 단일 출처다. 이 문서는 명령을 다시 적지 않고 그 전제(테스트 의도 표식·정적분석 경계·stub 한계 등)만 정리한다.

## 단위 테스트 프레임워크

- 어떤 단위 테스트 프레임워크를 쓰는지(예: XCTest, Swift Testing)는 이 마스터가 단정하지 않는다 — 프로젝트 프로필·`architecture.md`가 공급한다. 미공급이면 사용자에게 묻거나 (가정)으로 둔다.
- **테스트 TODO**: 구현 전 테스트 의도는 코드에 표식한다 — 표식 메커니즘은 프로젝트가 쓰는 프레임워크를 따른다(예: skip 마커 + TODO 메시지, 또는 미구현 test 명명 규약). IMPL 종료 시 "TODO 표식 → 구현된 test 전환"을 grep 커버리지 게이트로 쓴다 (자동화 단위 테스트 한정, 수동 동작 테스트와 별개).
- 테스트 타깃 구성(테스트 번들이 모듈별로 선언되는지, 앱레벨 테스트 유무 등)은 프로젝트의 모듈 시스템 정의를 따른다. SPM이면 각 패키지의 `Package.swift`가 테스트 타깃을 선언하는 형태가 한 예다.

## 스냅샷 테스트 (가정)

- 스냅샷 테스트 채택 여부·라이브러리는 (가정). 채택 시 동작 규약:
  - **검증 단계**: 스냅샷 테스트는 **오라클**로 동작한다 — 기준 이미지와 diff가 나면 **fail**. 사람이 개입하지 않는다.
  - **record 단계**: 기준 이미지를 새로 굽거나 갱신하는 record 모드에서만 사람이 개입해 결과를 검토·승인한다.
- 즉, 스냅샷 diff를 "통과로 간주"하거나 검증 단계에서 자동 record로 덮어쓰지 않는다. record는 별도 단계로 분리한다.

## UI 테스트 (가정)

- UI 테스트(예: XCUITest) 채택 여부·도구는 (가정) — 프로젝트가 공급한다. 채택 시 단위 테스트와 별도 타깃·별도 CI 잡으로 분리한다.

## CI (가정)

- CI 파이프라인(어떤 잡이 테스트를 돌리는지, 스냅샷·UI 테스트가 어느 잡에 묶이는지)은 (가정) — `.github/workflows/` 등 실측 후 확정.
- 정적분석(SwiftLint·SwiftFormat) 잡과 테스트 잡의 분리 여부도 (가정).

## 정적분석과의 경계 (확정 방향)

테스트로 거르기 전에 정적분석이 먼저 거른다 (`deploy/rules/global.md` 「자동화 가능 요청은 메커니즘 역제안」과 같은 사다리).

- 네이밍·스타일·타입 단순 위반 → 린터·포매터(예: SwiftLint·SwiftFormat). 설정 파일 채택 여부는 프로젝트가 공급(미공급이면 가정).
- 선언하지 않은 import → 모듈 시스템의 컴파일이 잡는다. 의존 그래프에 없는 모듈을 import하면 빌드 실패한다 (이것만). SPM이면 `Package.swift`의 `dependencies`가 그 그래프다.
- 레이어 역방향 의존(단방향 의존 위반) → 컴파일은 못 잡는다. 의존 그래프에 역방향을 적으면 import도 컴파일도 통과하므로, 각 모듈의 의존 선언을 레이어 규칙(프로젝트 `layer-rules.json`)과 정적 검사·grep으로 대조해 잡는다.
- 위 둘은 서로 다른 검사이므로 혼동하지 않는다.

## stub 검증의 한계 (확정)

stub(시그니처만 만들고 본문은 미구현)으로 구조를 세울 때, 컴파일러가 보증하는 범위와 못 잡는 범위를 구분한다.

- **컴파일러가 보증**: 시그니처 정합(타입·파라미터·반환형·프로토콜 준수). 호출부와 선언부가 맞는지.
- **컴파일러가 못 잡음**: 미구현 누락(빈 함수). 본문이 비어 있어도, 또는 `return` 한 줄만 있어도 시그니처만 맞으면 컴파일된다.
- **`fatalError`로 못 메우는 것**: 저장 프로퍼티의 초기값. 함수 본문은 `fatalError("not implemented")`로 스텁할 수 있지만, 저장 프로퍼티는 초기값을 가져야 컴파일되므로 `fatalError`로 미구현을 표시할 수 없다. 따라서 "미구현 프로퍼티"는 컴파일러가 강제로 드러내지 못한다 — 테스트나 리뷰가 잡아야 한다.

## Observation·동시성 전제

- Observation(예: `@Observable`) 사용 여부는 프로젝트의 UI 프레임워크·배포 타깃 정의를 따른다 — 이 마스터가 단정하지 않는다. 가용 여부(예: 매크로의 최소 OS 요건)는 프로젝트 프로필의 배포 타깃이 1차 출처다.
- 동시성 검사가 "컴파일러 강제"인지(예: Swift strict concurrency mode) 여부는 프로젝트 프로필·`architecture.md`가 공급한다. 강제 모드가 켜져 있으면 데이터 경합 위반은 경고가 아니라 컴파일 에러가 되므로, 그 경우 동시성 위반은 빌드 게이트가 잡는 항목으로 다룬다. 미공급이면 (가정)으로 두고 단정하지 않는다.
