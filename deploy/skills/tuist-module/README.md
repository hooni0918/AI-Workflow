# Tuist Module

Tuist 기반 iOS 프로젝트에서 모듈을 추가하거나 재구성하는 스킬입니다. 그래프를 먼저 설계하고, 적용한 뒤, 빌드와는 별개로 그래프 규칙을 지켰는지 따로 검증합니다.

## 동기

모듈을 추가하다 보면 빌드가 초록불이 되는 순간 "다 됐다"고 착각하기 쉽습니다.

하지만 빌드 성공은 "내가 manifest에 적어 둔 의존 안에서 타입이 맞는다"는 사실일 뿐입니다. Feature가 옆 Feature를 직접 끌어와도, Domain이 Feature를 거꾸로 의존해도, 그 의존을 manifest에 적어 두기만 하면 컴파일러는 통과시킵니다. 레이어 단방향 규칙을 어겼는지는 빌드가 봐 주지 않습니다.

그래서 "빌드가 됐으니 구조도 맞겠지"라는 자기증명에 빠지지 않도록, 설계와 검증을 빌드에서 떼어 냈습니다.

## 접근 방식

작업을 두 세션으로 나눕니다.

**설계 세션(PLAN)**에서는 코드를 한 줄도 만들지 않습니다. 새 모듈을 어느 레이어에 둘지, 어느 방향으로만 의존하게 할지를 먼저 정하고, 그 결정을 Tuist manifest에 옮깁니다. 그래프를 그리는 것이 먼저이고, 파일 생성은 그다음입니다.

**적용 세션(APPLY)**에서는 모듈 디렉토리와 소스 골격을 만들고 빌드합니다. 그리고 빌드와는 **별개로** 두 가지를 따로 검사합니다. 하나는 소스가 import 하면서 manifest에는 안 적은 의존이 있는지(`tuist inspect`), 다른 하나는 실제 모듈 의존 위상이 정해 둔 레이어 방향을 거스르지 않는지(`tuist graph`)입니다. 두 검사는 잡는 대상이 다르므로 둘 다 돌립니다.

## 산출물

설계 세션에서는 그래프 설계서와 manifest 변경안이 나옵니다. 적용 세션에서는 모듈 골격과, 빌드 통과 + 두 갈래 검증 통과 기록이 나옵니다. 골격의 빈 함수 본체는 시그니처만 맞춘 것이라 기능 동작까지 보증하지는 않으며, 본체 구현은 사람이 이어서 진행합니다.

## 의의

모듈이 늘수록 의존 그래프는 사람이 머릿속으로 추적하기 어려워집니다. "빌드가 되니 괜찮다"는 신호는 구조가 무너지는 것을 가려 줍니다. 이 스킬은 그래프 규칙 준수를 빌드와 분리된 기계 검사로 만들어, 모듈이 늘어도 레이어 경계가 흐려지지 않게 합니다.
