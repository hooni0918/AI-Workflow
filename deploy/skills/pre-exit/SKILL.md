# 세션 마무리

## 목적

세션을 회고하여 다음 세션에서 AI가 더 잘 동작하도록 만든다 (문제 리스트업, 규칙화+사례 축적, 미커밋 확인).

핵심 원칙: 사용자가 지적한 것은 다음 세션에서 반복되지 않아야 한다.

## Step 0. 세션 자료 수집

### 0a. 팀 에이전트 회고

[CRITICAL] [team-agent](../../contexts/team-agent.md) 규칙을 따른다.

team-agent 규칙으로 띄운 활성 팀/서브에이전트가 있으면, 현재 런타임의 후속 지시 경로로 각 에이전트에게 세션 회고를 요청한다. 에이전트가 발견한 문제는 Step 1에 함께 포함한다. 수집 완료 후 현재 런타임에서 제공하는 종료/정리 수단으로 에이전트를 종료한다.

활성 팀 에이전트가 없으면 이 하위 단계를 건너뛴다.

## Step 1. 문제 리스트업

세션에서 **문제만** 추출한다. 잘한 것은 다루지 않는다 — 회고의 목적은 다음 세션의 사고 방지이므로, 포착할 대상이 없으면 "문제 없음"으로 보고하고 Step 2를 건너뛴다.

- 사용자가 지적하거나 반려한 것
- 사용자가 방향을 수정한 것
- 같은 유형의 실수가 반복됐거나 임팩트가 큰 것

각 문제를 한 줄로 요약하여 사용자에게 제시한다. 컴파일·린트·커밋 메시지·파일 구조 같이 기계적으로 검출 가능한 패턴이면 `(도구화 후보)` 태그를 함께 표시한다.

### 사용자 측 재작업 신호 점검

Step 1 본문은 AI 측 문제를 잡는다. 여기에 더해, 세션에 재작업·재지시가 있었으면 그 원인이 **사용자 요청 측**에 있었는지 점검한다. AI를 다시 시키게 됐다는 건 입력에 아래 중 하나가 빠졌다는 신호다.

- 기술스택 이해부족 — 어떤 라이브러리·버전·패턴 위에서 도는지 전제가 안 잡힘
- 코드구조 이해부족 — 기존 파일 배치·의존·관례가 요청에 안 담김
- 불명확한 설명 — 요구가 다의적이라 어떤 해석이든 절반은 빗나감
- 한 번에 과한 요청 — 한 커밋·한 검토 단위를 넘는 분량

해당하면 어느 신호였는지 사용자에게 짚어 보고한다. 이건 AI 규칙으로 심는 대상이 아니라 사용자 자기 회고용이므로 Step 2 규칙화는 스킵한다. 단, 같은 신호가 여러 세션 반복되면 그 패턴을 사용자에게 알린다.

### 제안 거부 신호 수집

AI가 제안했으나 사용자가 그대로 받지 않고 다르게 판단한 항목을 별도로 모은다. 이건 AI "실수"가 아니라 **사용자 선호 신호**다 — 다음 세션에서 처음부터 사용자가 원하는 방식으로 제안하기 위한 입력이다. (별도 로그 파일을 만들지 않는다. 세션 내용은 메인이 이미 알고 있으므로, 종료 시점에 직접 회수해 리스트업한다.)

각 항목을 `{내 제안 → 사용자 선택 → 드러난 선호}` 한 줄로 정리한다.

- 예: `도입부 5문장 전체 출제 제안 → 사용자가 앞 2문장만 지정 → 한 번에 좁은 범위씩, 사용자가 지정한 단락만`

이 항목들은 Step 2 규칙화 대상에 합류한다. 단, **반복될 선호로 판단되는 것만** 골라 해당 스킬(digest/explain/review 등)의 제안 규칙에 사례로 심는다. 일회성 변덕(그 세션 한정 사정)은 규칙화하지 않고 그 판단을 보고한다.

## Step 2. 규칙화 + 교정 사례 축적

Step 1의 각 문제에 대해 다음을 수행한다. 단, 세션 특유의 1회성 문제라 일반화가 부적절하면 규칙화를 스킵하고 그 판단을 보고한다.

### 2-1. 정적분석 보완 가능성 확인

규칙화 전에, 해당 문제를 정적분석·자동화 도구로 강제할 수 있는지 먼저 확인한다. 가능하면 프롬프트 규칙 대신(또는 함께) 도구 설정 변경을 제안한다.

- Swift 코드 패턴(강제 언래핑, force-try, 접근제어, 네이밍 등) → SwiftLint 룰 (`.swiftlint.yml`의 표준 룰·`custom_rules`)
- 포매팅(들여쓰기, import 정렬, 공백) → SwiftFormat
- 모듈 의존 방향·미선언 import → Tuist 그래프 검사 (`tuist inspect`·`tuist graph` — [conventions/tuist.md](../ios-workflow/conventions/tuist.md))
- 커밋 메시지 형식·범위 → git `commit-msg` hook (`scripts/commit_msg.py`, `make install-hooks`로 설치). scope-enum에 항목 추가도 이 스크립트에서 한다
- 파일 구조·금지 경로 → 정책 hook (`deploy/hooks/`), CI lint

도구로 잡히는 것은 도구로, 의미·맥락 판단이 필요한 것만 프롬프트 규칙으로 남긴다. 양쪽 다 필요한 경우(도구가 부분만 잡는 경우) 보완 관계를 명시한다.

### 2-2. 분류

문제의 성격에 맞는 맥락을 특정한다:

- 코딩 컨벤션 문제 → `deploy/contexts/coding-standards/`
- 테스트·CI·검증 전제 문제 → `deploy/contexts/testing-strategy/assumptions.md`
- 스킬 동작 문제 → 해당 스킬의 SKILL.md 또는 specialized 체크리스트
- 워크플로우 문제 → 해당 워크플로우 스킬
- 일반 AI 동작 교정 → `deploy/rules/`

어느 분류에도 깔끔히 맞지 않으면 규칙화를 강행하지 말고 사용자에게 그 사실을 알린다.

### 2-3. 규칙 문안

상황과 원하는 행동을 담은 규칙 초안을 작성한다. 부정형("~하지 마세요")·긍정형("~하세요") 중 상황에 자연스러운 쪽을 택한다.

### 2-4. 교정 사례

규칙에 대응하는 before/after를 한 쌍으로 작성한다:
- **before** — 이번 세션에서 실제로 발생한 잘못된 형태
- **after** — 규칙을 적용해 교정한 형태

사례는 규칙과 같은 맥락에 둔다 (예: coding-standards 규칙이면 coding-standards에 사례도 함께).

### 2-5. 승인

사용자에게 위치·규칙·사례를 제시하고, 승인받은 것만 반영한다.

## Step 3. 미커밋 변경사항 확인

`git status`로 커밋되지 않은 변경사항이 있는지 확인한다. 있으면:
- 파일 목록을 사용자에게 보고한다
- 커밋 여부를 묻고, 사용자가 지시하면 이어서 커밋까지 수행한다
- 지시가 없으면 그대로 둔다 (다른 세션의 작업이거나 사용자가 의도적으로 미커밋 상태로 둔 것일 수 있으므로, 임의로 커밋하지 않는다)

## Step 4. 워크트리 정리

이번 세션에서 `git worktree add`로 생성한 워크트리만 대상이다. 이전 세션부터 존재하던 워크트리는 그대로 두고 언급하지도 않는다.

대상 워크트리에 대해:
- 미커밋 변경이 없으면 사용자 지시를 기다리지 않고 `git worktree remove`로 제거한다
- 미커밋 변경이 있으면 파일 목록을 보고하고 지시를 기다린다
