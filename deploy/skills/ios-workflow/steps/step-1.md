# Step 1: 배경 파악 및 문제 정의

> **이 단계의 목표: 작업의 문제/목표를 정의한다** (도메인/제품 관점, 코드 분석 아님)

step-1은 BG 세션 안에서 두 sub-step으로 진행된다. 1.1은 자료 입수 단계, 1.2는 requirement-review 본체.

- **Step 1.1 자료 받기**: 사용자에게서 자료를 받아 정리. 자료 입수 완료 시점이 후속 세션(MARKUP)의 spawn 가능 조건이다.
- **Step 1.2 requirement-review 본체**: 화면별 엣지케이스 선제 질문 + 중복 조기 발견 + 작업 익숙도 판별

---

## Step 1.1 자료 받기

### 자료 수집 점검

외부에서 시안·기획서·Figma 자료를 받는 케이스라면, 자료 수집 방식이 적절한지 점검한다. 자료를 아직 받기 전이면 능동 안내, 이미 받았는데 흐릿하면 영역을 짚어 다시 요청한다.

**step-1.1은 자료의 단일 수집 지점이다.** 기획·요구사항과 함께 **사용 가능한 figma/시안 자료를 여기서 전부** 받아 저장한다 (figma URL·캡처 → `retained/figma-url.md`·`retained/figma/`). 다운스트림(requirement-review·MARKUP)은 이 수집분을 *참조*하며 자료를 재요청하지 않는다.

**개인 모드**는 외부에서 받을 자료가 없다 (figma 없음, 기획자·디자이너 없음). \"받기\" 대신 사용자와 함께 기획·디자인 스펙을 **정의**하는 것이 이 단계의 일이며, 정의는 step-1.2 requirement-review에서 screen.md로 수행한다. step-1.1에서는 사용자가 이미 가진 참고 자료(디자인시스템 소스, 기존 코드 등)만 수집한다.

### 컨벤션 소스 수집 — 이름 스캔 선제안 + conventions-index.md

프로젝트 컨벤션 소스 수집의 **단일 지점**이다. 이후 단계는 여기서 만든 인덱스를 소비만 하고 재수집·재질문하지 않는다.

1. **이름 스캔 선제안**: 작업 레포가 존재하면 Glob으로 컨벤션 성격의 파일을 **이름만** 스캔한다 — `**/CLAUDE.md`, `**/AGENTS.md`, `*.swiftlint.yml`, `**/Package.swift`, `*.xcodeproj/project.pbxproj`, `docs/**`, `*ARCHITECTURE*`, `*convention*` 류. 후보 목록을 사용자에게 선제안한 뒤 채택받는다. **이 단계에서 파일 내용은 읽지 않는다** (컨텍스트 오염 방지).
2. **사용자 보충**: \"추가로 참고할 자료가 있나요? (예: 사내 컨벤션 가이드, 디자인시스템, 기존 기술 스택 제약 등)\" 사용자 제공 원문은 `/plan/background/retained/`에 저장하고 인덱스에 등재한다.
3. **인덱스 작성**: 채택된 소스를 `/plan/background/retained/conventions-index.md`에 기록한다. 양식은 항목당 **\"경로 + 한 줄 트리거(어떤 작업·모듈에서 봐야 하는지)\"**. 본문 요약은 적지 않는다.
   - **빌드/검증 명령 단일 출처 등재**: [conventions/spm.md](../conventions/spm.md)를 인덱스에 등재한다. 트리거: \"빌드·테스트·SwiftLint·레이어 의존성 검사 명령이 필요할 때\". [가정] 도구 환경(테스트 프레임워크·CI)이 실제와 다르면 그 차이를 인덱스에 한 줄로 남긴다.
4. **레포가 아직 없으면** 스캔·인덱스 생성을 레포 확보 시점으로 미룬다.

### Step 1.1 종료 — 분기점

step-1.1 종료 = `persistent/`에 원본 자료 저장 + 사용 가능한 figma/시안 전체를 `retained/figma-url.md`·`retained/figma/`에 저장 + 컨벤션 인덱스 산출. 그 이상(분석)은 모두 step-1.2 영역이다.

step-1.1은 후속 세션 spawn 가능 분기점이다. 자료 저장 직후, **분석(step-1.2)으로 진입하기 전에** ① 컨벤션 인덱스 산출, ② SKILL.md 「세션 spawn 안내 메커니즘」 발동. 이 둘을 내지 않은 채 분석·구현을 시작하지 않는다. ①을 건너뛰고 다운스트림으로 가면 컨벤션 1차 소스를 거기서 재수집하게 되고, 같은 추출 패턴의 Reviewer가 그 누락을 못 잡는 자기증명 루프가 생긴다.

## Step 1.2 requirement-review 본체

### 화면 종류별 엣지케이스 선제 질문

iOS 화면은 기획서가 잘 다루지 않는 빈틈이 많다. 각 화면을 아래 **공통 iOS 엣지케이스**에 대고 검토하고, 정의되지 않은 항목을 사용자에게 선제 질문한다. 화면 종류별 추가 체크리스트는 해당 화면에 함께 적용한다.

**공통 iOS 엣지케이스 (모든 화면)**:
- **다크모드**: 라이트/다크 양쪽에서 색·이미지·그림자가 정의되어 있는가? 에셋 카탈로그에 다크 변형이 있는가?
- **Dynamic Type**: 시스템 글자 크기를 최대(접근성 크기 포함)로 키웠을 때 레이아웃이 깨지지 않는가? 줄바꿈/말줄임/스크롤 처리가 정의되어 있는가?
- **SafeArea / 노치 / Home Indicator**: 상·하단 SafeArea를 무시할 영역(배경)과 지킬 영역(콘텐츠·버튼)이 구분되어 있는가?
- **빈 상태**: 데이터가 0건일 때 화면이 정의되어 있는가? (아래 리스트 화면 체크리스트와 연동)
- **딥링크**: 외부에서 이 화면으로 직접 진입할 수 있는가? 선행 데이터 없이 진입하면 어떻게 되는가?
- **백그라운드 복귀**: 백그라운드에 갔다가 복귀(scene 활성화)하면 데이터 갱신·타이머·인증 만료 처리가 정의되어 있는가?
- **회전 / 멀티태스킹**: 가로/세로 회전, iPad Split View에서 레이아웃이 어떻게 바뀌는가? 회전 지원 여부가 정해져 있는가?

화면 종류별 추가 체크리스트:
- 리스트형: [requirement-review/screen-type/list.md](../requirement-review/screen-type/list.md)
- 상세형: [requirement-review/screen-type/detail.md](../requirement-review/screen-type/detail.md)
- 폼형: [requirement-review/screen-type/form.md](../requirement-review/screen-type/form.md)
- 순차 플로우: [requirement-review/screen-type/sequential-flow.md](../requirement-review/screen-type/sequential-flow.md)

**진행 방식**: AI는 요구사항을 비판적으로 분석하고, 발견한 빈틈을 질문으로 제시한다. 단정하지 않고 장단점을 함께 설명한다. 화면마다 같이 분석 → `/discussion`으로 미결 항목 결론 → 산출물 확정. 확정 내용은 `pr{N}/consumable/screen.md`(화면별)·`background/`(공통)에 기록하고, 사용자가 판단할 수 없는 항목은 TODO로 분류한다.

### 중복 조기 발견

requirement-review 단계에서 디자인·컴포넌트 중복이 있으면 **사용자에게 명시 통보**한다. 분석 산출물에 묻어서 흡수만 시키지 않는다. 형식: \"시안 카드 N종에서 공통 골격 발견 → 컴포넌트 추출 대상. 지금 통합 결정 vs 나중에 PLAN으로 미루기 중 선택\".

### 작업 익숙도 판별

사용자에게 \"이번 작업에서 처음 다루는 기술이나 낯선 영역이 있나요?\"를 확인한다. 낯선 작업이라면:

1. 공부 세션을 위한 인수인계 문서를 작성한다 (왜 필요한지 / 알아야 하는 키워드·핵심 API를 학습 순서대로). 구체적 학습 방법(how)은 공부 세션에서 결정.
2. **별도 세션에서 인수인계 문서 기반으로 병행 학습**하면서, 이 세션에서는 기획을 계속 진행하도록 안내.
3. 사용자는 방향 잡기(라이브러리 선택, 참조할 원본 코드 지정)에 집중, 구현 판단은 AI가 맡되 리뷰를 더 꼼꼼히 받는 구조.

인수인계 문서가 작성되었으면 PR_{N}_PLAN 진입 안내 시 \"학습 완료 뒤 진입하라\" 조건을 함께 안내한다 (SKILL.md 「학습 인수인계 후 세션 진입 대기」).
