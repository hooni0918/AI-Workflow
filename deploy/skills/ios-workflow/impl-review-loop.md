# 구현↔리뷰 루프 (iOS)

## 목적

구현 대상을 채우고 [진실원천 충실도 + 코딩 스탠다드] 두 축으로 0건까지 수렴시킨다.

호출자가 **[구현자 · 재료 · 진실검사 A · 규칙검사 B · 증분 단위]**를 주입하면, 본 문서는 그 위에서 "구현 → 두 축 검증 → 0건 수렴"을 돌린다. 구현 종류(MARKUP / 로직 IMPL)에 무관하게 골격은 동일하며, 종류 차이는 주입 인자에만 담긴다. 워크플로우의 구현 성격 세션 절차(step 또는 세션 본문)가 인자를 주입해 호출한다. 독립 호출 없음.

팀 운용은 현재 런타임이 제공하는 에이전트 기능 기준(Claude Code면 Agent Teams), 규칙검사 리뷰 절차는 [code-review](../code-review/SKILL.md)를 따른다.

## 입력 (호출자가 주입)

| 인자 | 의미 | 비고 |
|---|---|---|
| 구현자 | 증분을 만드는 주체 (+ 모델). 1명 이상 | 종류별로 다름 |
| 재료(context) | 무엇을·어떻게 만들지 — 계획·결정·stub·진실 원천 등 구현 입력 일체 | AI 산출물 포함 |
| **진실검사 A** | (진실 원천 핸들 · 증분 통과기준 · 종료 커버리지 기준 · 검증 메커니즘) | **필수·non-null** |
| 규칙검사 B | 적용할 코딩 스탠다드 묶음 (SwiftLint 포함) → Reviewer 구성 | 양쪽 공통 골격, 내용물만 주입 |
| 증분 단위 | 한 번에 만들고 검사·커밋하는 변경 묶음 | 종류별로 다름 |

**재료 ≠ 진실검사 A.** 진실 원천 자체는 재료에도(구현 시 참조) 진실검사 A에도(검사 기준) 들어간다. 그러나 AI가 만든 산출물은 재료에만 들어가고 A의 검사 기준이 될 수 없다 — 자기증명 루프 차단([SKILL.md 「검증 기준 = 진실 원천」](SKILL.md)).

## 두 세션의 주입 예

| 세션 | 구현자 | 진실검사 A (메커니즘) | 규칙검사 B | 증분 단위 |
|---|---|---|---|---|
| MARKUP (실무) | Markup Implementer | figma 원본 직접 fetch ↔ 마크업 대조. **대조형**(실행 오라클 없음 — 디자인은 돌려서 판정 불가). 종료 커버리지: figma 화면·상태·토큰 전수 | SwiftLint + 마크업 컨벤션 | 화면/컴포넌트 단위 |
| MARKUP (개인) | Markup Implementer | 사용자 시각 확인 ↔ 기획 md 대조. **대조형** (figma 없음, 사람이 진실 원천) | SwiftLint + 마크업 컨벤션 | 화면/컴포넌트 단위 |
| 로직 IMPL (step-5) | Feature Implementer | 테스트 실행 green(swift test/xcodebuild test) + 테스트 TODO 커버리지. **오라클형**(실행이 곧 판정) | SwiftLint + 로직 컨벤션 + Advanced | 로직 커밋 |

## 두 검증 축

### A. 진실원천 충실도 (non-null · 매 증분 게이트)

- 주입된 진실 원천을 **직접 fetch/실행**해 증분과 대조한다. 캐시·AI 산출물을 검사 기준으로 쓰지 않는다.
- 증분이 통과 기준에 미달이면 그 증분은 통과하지 못한다. **어떤 호출도 축 A 없이 종료할 수 없다.**
- 검증 메커니즘은 진실 원천의 성질로 주입된다: 실행 오라클(테스트)이 있으면 그 실행 결과로, 없으면(디자인) Reviewer ↔ 구현자 직접 대조 루프로.
- **팀 구성은 A 메커니즘의 함수다.** 오라클형(로직)이면 축 A Reviewer를 spawn하지 않는다 — 테스트 실행 결과가 곧 게이트다. 대조형(마크업)이면 Reviewer ↔ 구현자 직접 루프를 spawn한다.
- 모든 증분 종료 후 주입된 **종료 커버리지 기준**으로 누락 0건을 1회 점검한다.

### B. 코딩 스탠다드 (공통 루프)

- 주입된 스탠다드로 Reviewer ×N + 자유 리뷰를 0건까지 돌린다. SwiftLint는 `swiftlint lint --strict`를 직접 실행하는 기계 판정 ([conventions/spm.md](conventions/spm.md)). Coding-Standards 지적은 근거 컨벤션 파일 경로를 동반한다. Reviewer 결과는 Lead가 종합한다 — 중복 제거, sonnet 오탐 필터링, 검증된 이슈만 구현자에게 한번에 전달.
- **축 B는 진실원천 충실도를 판정하지 않는다.** 충실도 의심을 발견하면 스스로 정답 처리하지 말고 **축 A로 넘긴다.**

## 루프

증분 단위마다:
1. 구현자가 증분을 만든다.
2. 두 축으로 검증한다 — 각 축 **0건까지** (미달 시 구현자 수정 → 재검).
   - 두 축의 순서·병렬은 주입된 **A 메커니즘의 성질**(기계 사실)로 정해진다. A가 **대조형**(마크업)이면 A를 먼저 0건 수렴시킨 뒤 B를 돌린다 — figma를 오독해 어긋난 증분을 B가 헛리뷰하는 것을 막는다. A가 **오라클형**(로직)이면 A·B 병렬.
3. 다음 증분으로.

모든 증분 종료 후 **종료 커버리지 점검**(축 A). 누락 발견 시 해당 증분 흐름으로 복귀.

## 스냅샷 테스트 (있는 경우) [가정]

프로젝트가 스냅샷 테스트를 쓰면 마크업의 진실 원천 보조 오라클로 활용할 수 있다. 단 단계별 역할이 갈린다:

- **record 단계 (baseline 생성)**: 사람 개입 필수. 사용자가 baseline 이미지를 figma 원본(개인 모드는 디자인 의도) 기준으로 직접 검수·확정한다. AI가 baseline을 자동 정답 처리하지 않는다 — 잘못된 baseline은 자기증명 루프의 시작점.
- **검증 단계 (회귀 방지)**: baseline 확정 후에는 오라클로 동작한다. diff 발생 시 **fail** — 의도된 변경이면 사용자가 record를 다시 트리거하고, 아니면 마크업을 수정한다.

스냅샷 도입 여부·도구는 프로젝트마다 다르므로 [가정] 라벨로 둔다. 도입돼 있으면 위 record/검증 분리를 따르고, 없으면 대조형 A는 Reviewer ↔ 구현자 직접 대조로 진행한다.

## 우회 (호출자의 사전 점검 — 입력 아님)

다음 두 가지가 **모두 참**일 때만 호출자가 본 엔진을 생략할 수 있다. 둘 다 **기계 판정**:
1. 구현 전 계획에서 **진실원천 아티팩트가 선언되지 않음** (파일 부재로 검증 가능)
2. **변경 파일 전부가 프로젝트 자동 검사 도구 스코프 안** (SwiftLint glob·빌드 대상 — 스코프 매칭으로 검증 가능)

하나라도 거짓이면 엔진 강제. 의견 기반 생략("간단해 보임")은 차단한다.

## 출력

- 워크트리에 **검증 완료된 커밋들** — 모든 증분이 A·B 0건으로 수렴.
- 호출자에게 **수렴 보고** — 증분별 라운드 결과 + 종료 커버리지 점검 + 수정 사항.

산출물 파일을 만들지 않는다 (리뷰 결과 정리가 필요하면 호출자 책임).
