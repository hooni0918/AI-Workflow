# iOS Workflow

제가 iOS 작업을 하는 방식을 프롬프트로 옮긴 것입니다. 기획서, 피그마 디자인, 버그 리포트 같은 자료를 넣으면 PR이 나옵니다.

## 동기

Quriously에서 iOS를 개발하면서, 기획서를 받았다고 바로 코드를 짜면 배경을 잘못 이해한 채 작업이 굴러가거나 한 PR이 너무 커져서 리뷰가 어려워지는 일을 반복했습니다. 그래서 저는 먼저 배경을 파악하고, 작업을 PR 단위로 나누고, 구현 방침을 세운 뒤에 코드를 작성합니다. 구현이 끝나면 빌드와 테스트로 점검하고, 리뷰어가 읽기 좋은 PR 본문을 씁니다.

이 순서를 프롬프트로 옮겨서 AI도 같은 방식으로 일하도록 만든 것이 이 워크플로우입니다.

또 하나의 목표는 작업을 가능한 한 병렬로 돌려 기다리는 시간을 줄이는 것입니다. 다음 PR의 설계와 현재 PR의 구현을 겹치고, 화면 마크업은 PR 작업과 동시에 진행하고, 리뷰어를 종류별로 나눠 동시에 돌립니다.

## iOS 작업이라 먼저 묻는 것들

iOS 화면은 기획서가 잘 다루지 않는 빈틈이 많습니다. 다크모드에서 색이 어떻게 보이는지, Dynamic Type을 키웠을 때 레이아웃이 깨지지 않는지, SafeArea와 노치를 어떻게 다루는지, 데이터가 0건일 때 빈 상태 화면이 있는지, 딥링크로 중간 화면에 바로 진입하면 어떻게 되는지, 백그라운드에 갔다가 복귀하면 상태가 유지되는지, 가로/세로 회전 시 무엇이 바뀌는지 — 이런 것들을 코드를 짜기 전에 질문으로 먼저 끄집어냅니다.

화면 종류(리스트·상세·폼·순차 플로우)마다 빠지기 쉬운 엣지케이스가 다르므로, 종류별 체크리스트를 두고 요구사항 리뷰 단계에서 선제 질문합니다.

## 비슷한 코드를 먼저 찾습니다

새 화면이나 새 로직을 만들기 전에, 기존 코드베이스에서 같은 역할을 하는 best-practice 패턴을 먼저 찾습니다. 이미 있는 ViewModel 구조, 네트워크 레이어, 재사용 컴포넌트를 참조 링크로 걸어 두고 그 위에서 구현합니다. 같은 것을 두 번 만들지 않기 위해서입니다.

## 만들고, 검사하고, 0건까지 고치는 엔진

화면 마크업을 만드는 작업과 로직을 구현하는 작업은 겉보기엔 다르지만 골격이 같습니다. "만들고 → 정답과 대조하고 → 틀린 게 0건이 될 때까지 고친다"는 뼈대를 공유합니다. 다른 것은 무엇을 정답으로 보느냐뿐입니다. 마크업의 정답은 피그마 원본(개인 작업이면 사용자가 눈으로 확인하는 디자인 의도)이고, 로직의 정답은 실제로 동작하는지 — 즉 테스트(swift test/xcodebuild test)가 통과하는지입니다.

그래서 이 반복 과정을 하나의 엔진으로 빼고, 두 작업이 정답과 검사 기준만 바꿔 끼워 같이 씁니다.

이 엔진은 여러 AI를 한 팀으로 묶어 돌립니다. 만드는 역할과 검사하는 역할을 나눠서 만든 사람이 직접 검사하지 않게 합니다.

- 만드는 역할: UIKit 마크업(UIViewController+AutoLayout) 또는 로직을 작성하고 작업을 저장합니다.
- 정답 대조 역할: 마크업은 피그마 원본을 직접 열어 대조하고(개인 작업은 사용자 시각 확인), 로직은 테스트(swift test/xcodebuild test)를 실제로 돌려 동작을 확인합니다.
- 코딩 규칙 검사 역할: SwiftLint를 직접 돌리고 정해 둔 코딩 규칙 문서를 읽어 어긋난 곳을 찾습니다.
- 깊이 있는 품질 검사 역할: 더 무거운 판단이 필요한 코드 품질을 살핍니다.
- 지휘 역할: 사용자와 소통하고 검사 결과를 모읍니다.

만드는 역할이 작업을 저장하면 검사 역할이 살펴보고, 어긋난 점이 있으면 만든 역할에게 다시 돌려보냅니다. 이 주고받기를 어긋난 점이 0건이 될 때까지 반복합니다.

## 설계 근거

### 만든 사람과 검사하는 사람을 나눈 이유

코드를 작성한 쪽이 직접 검사하면, 자기가 잡은 방향에 치우쳐서 문제를 놓치기 쉽습니다. 독립된 검사자가 처음 보는 시선으로 살펴야 실제 문제를 잡아냅니다.

### 검사자를 여러 명으로 나눈 이유

먼저 정확도입니다. 지켜야 할 코딩 규칙이 많아서 한 명이 전부 동시에 보면 놓치는 게 생깁니다. 종류별로 나누면 각자 맡은 부분에만 집중할 수 있습니다.

다음은 비용입니다. 기계적으로 대조하는 일은 가벼운 모델에, 깊은 판단이 필요한 검사는 무거운 모델에 맡깁니다.

### 0건까지 반복하는 이유

리뷰에서 찾은 문제를 고친 뒤 다시 확인하면, 여전히 문제가 남아 있는 경우가 있습니다. 한 번 검사로는 충분하지 않기 때문에, 문제가 0건이 나올 때까지 검사와 수정 사이클을 돌립니다.

### 검사 기준은 항상 원본

검사할 때 기준으로 삼는 것은 언제나 원본입니다. 마크업은 피그마 원본 화면, 로직은 실제 테스트 결과, 규칙은 코딩 규칙 문서 그 자체입니다.

AI가 중간에 만들어 둔 정리본을 검사 기준으로 쓰지 않습니다. 그 정리본은 만들 때 빠뜨린 부분이 있을 수 있고, 같은 빠뜨림을 검사자도 똑같이 못 보고 지나치면 문제가 그대로 남기 때문입니다. 정리본은 원본을 빠르게 찾기 위한 메모로만 쓰고, 검사는 항상 원본을 다시 펼쳐 확인합니다.

## 세션 라인업

작업을 여러 세션으로 나눠 각각 별도의 맥락에서 진행합니다. 한 세션이 끝나면 다음 세션을 시작하라는 안내가 나옵니다.

- **BG** (배경 파악과 PR 분할): 기획서, 피그마, 버그 리포트를 읽고 무엇을 왜 해야 하는지 정리한 뒤, 작업을 PR 단위로 쪼갭니다. 모든 작업의 출발점입니다.
- **MARKUP** (화면 마크업): 피그마 시안을 화면, 구역, 작은 부품 단위로 잘라 UIKit 마크업(UIViewController+AutoLayout)을 만들고, 피그마 원본과 0건까지 대조해 검증합니다. 개인 작업이면 사용자가 직접 눈으로 확인합니다. 검증을 마친 마크업은 각 PR이 그대로 가져다 씁니다.
- **PR_{N}_PLAN** (PR 설계): 그 PR이 무엇을 하는지 정의하고 구현 방침을 정합니다. 설계 끝에서 다음 PR이 기댈 자리(공개 시그니처 stub)만 미리 잡아 두면, 다음 PR 설계와 이번 PR 구현이 동시에 굴러갑니다.
- **PR_{N}_IMPL** (PR 구현): 검사 엔진을 돌려 로직을 채웁니다. 화면 마크업은 MARKUP에서 검증한 것을 그대로 가져오므로 이 세션은 로직에 집중합니다.
- **PR_{N}_WRITING** (PR 본문 작성): 구현 맥락 없이 남은 산출물 파일만 보고, 리뷰어가 읽기 좋은 PR 본문을 씁니다.

아래는 자료를 넣어 PR이 나오기까지의 큰 흐름입니다. 옆으로 갈라지는 가지는 동시에 진행되는 작업입니다.

```
BG · 배경 파악 + 할 일을 PR 단위로 분할
  │
  ├──→ MARKUP · 화면 마크업 (PR 작업과 동시 진행)
  │
  ▼
PR_{N}_PLAN 설계 → PR_{N}_IMPL 구현·검사 → PR_{N}_WRITING 본문 작성 → 완성
  │
  └──→ 다음 PR_{N+1}_PLAN 설계는 PR_{N}_IMPL이 도는 동안 미리 시작
```

## 두 가지 모드

- **실무**: 회사 프로젝트. 기존 코드 위에서 시작하고, 피그마 시안을 디자인 진실 원천으로 씁니다.
- **개인**: 개인 프로젝트. 기획자·디자이너가 없어 피그마가 없습니다. 사용자와 함께 기획·디자인을 정의하고, 마크업 검증은 피그마 대조 대신 사용자가 직접 눈으로 확인합니다.
