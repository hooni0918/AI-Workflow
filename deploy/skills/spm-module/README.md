# SPM Module

SPM 로컬 패키지 기반 iOS 프로젝트에서 모듈을 추가하거나 재구성하는 스킬입니다. 의존 구조를 먼저 설계하고, 적용한 뒤, 빌드와는 별개로 레이어 규칙을 지켰는지 따로 검증합니다.

## 동기

모듈을 추가하다 보면 빌드가 초록불이 되는 순간 "다 됐다"고 착각하기 쉽습니다.

하지만 빌드 성공은 "내가 Package.swift의 dependencies에 적어 둔 의존 안에서 타입이 맞는다"는 사실일 뿐입니다. 같은 레이어 모듈이 옆 모듈을 직접 끌어와도, 하위 레이어가 상위 레이어를 거꾸로 의존해도, 그 의존을 dependencies 배열에 적어 두기만 하면 컴파일러는 통과시킵니다. 레이어 단방향 규칙을 어겼는지는 빌드가 봐 주지 않습니다.

그래서 "빌드가 됐으니 구조도 맞겠지"라는 자기증명에 빠지지 않도록, 설계와 검증을 빌드에서 떼어 냈습니다.

## 접근 방식

작업을 두 세션으로 나눕니다.

**설계 세션(PLAN)**에서는 코드를 한 줄도 만들지 않습니다. 새 모듈을 어느 레이어에 둘지, 어느 방향으로만 의존하게 할지를 먼저 정하고, 그 결정을 Package.swift에 옮깁니다. 의존 구조를 그리는 것이 먼저이고, 파일 생성은 그다음입니다.

**적용 세션(APPLY)**에서는 패키지 디렉토리와 소스 골격을 만들고 빌드합니다. 그리고 빌드와는 **별개로** 두 가지를 따로 검사합니다. 하나는 소스가 import 하면서 dependencies에는 안 적은 의존이 있는지(이건 SPM이 컴파일 실패로 잡습니다), 다른 하나는 각 Package.swift의 dependencies 배열이 정해 둔 레이어 방향을 거스르지 않는지입니다. layer-rules.json과 기계 대조하는 정적 검사기(`check_spm_layers.py`)가 위반 시 빌드를 막고, grep 대조는 보조로 씁니다. 역방향을 dependencies에 적으면 컴파일은 통과하므로, 배열 자체의 대조가 따로 필요합니다.

## 산출물

설계 세션에서는 의존 구조 설계서와 Package.swift 변경안이 나옵니다. 적용 세션에서는 모듈 골격과, 빌드 통과 + 두 갈래 검증 통과 기록이 나옵니다. 골격의 빈 함수 본체는 시그니처만 맞춘 것이라 기능 동작까지 보증하지는 않으며, 본체 구현은 사람이 이어서 진행합니다.

## 의의

모듈이 늘수록 의존 그래프는 사람이 머릿속으로 추적하기 어려워집니다. "빌드가 되니 괜찮다"는 신호는 구조가 무너지는 것을 가려 줍니다. 이 스킬은 레이어 규칙 준수를 빌드와 분리된 dependencies 대조로 만들어, 모듈이 늘어도 레이어 경계가 흐려지지 않게 합니다. SPM 로컬 패키지 레이어(프로젝트가 정의)의 정확한 목록·경계·방향은 프로젝트 프로필/architecture.md/layer-rules.json이 1차 출처입니다. 없으면 사용자에게 묻거나 안전 가정(가정 라벨)으로 진행합니다.
