# 모듈 레이어 — 경계와 의존 방향

레이어 배치·허용 의존 방향을 다루는 **방법론** 문서다. PLAN의 배치·방향 설계와 APPLY의 track B 검증이 모두 이 문서의 원칙을 기준으로 한다.

구체 레이어 목록·각 레이어의 책임·허용 의존표는 마스터가 고르지 않는다. 그 값은 **프로젝트의 `layer-rules.json`(기계 대조용)과 `architecture.md`(산문 정의)가 1차 출처**다. 없으면 사용자에게 묻거나 안전 가정(가정 라벨)으로 진행한다. 구성 예시는 [coding-standards/examples/clean-architecture-layer-rules.md](../../../contexts/coding-standards/examples/clean-architecture-layer-rules.md)에 있다 — 그 레이어 구성은 한 프로젝트가 고른 예시일 뿐 마스터 표준이 아니다.

## 레이어 모델

다층 모듈은 레이어를 위→아래로 쌓고, 의존이 한 방향으로만 흐른다. 위는 더 자주 바뀌는(휘발성 높은) 쪽, 아래는 더 안정적인(많이 의존받는) 쪽이다.

- 어떤 레이어가 몇 개 있고 각자 무엇을 책임지며 무엇을 의존할 수 있는지는 프로젝트가 정의한다 (`layer-rules.json`·`architecture.md`).
- 실제 레포의 레이어·의존 edge는 `Packages/` 디렉토리 구성과 각 `Package.swift`의 `dependencies` 배열로 확인한다.
- 이 문서는 그 레이어들 **사이에 항상 성립해야 하는 위상 원칙**과 그것을 검증하는 방법만 담는다.

## 허용 의존 방향 — 위상 원칙

### 1. 단방향 (상위 → 하위만)

- 의존은 휘발성 높은 상위에서 안정적인 하위로만 흐른다. 각 모듈은 프로젝트 규칙이 허용한 하위 레이어만 의존한다.
- **하위는 상위를 의존하지 않는다.** 안정적인 레이어가 자주 바뀌는 레이어를 향하는 역방향 edge는 모두 금지.

### 2. 수평 의존 금지

- 같은 레이어 모듈끼리 직접 의존하지 않는다 (예: 한 Feature 모듈이 다른 Feature 모듈을 import).
- 공유가 필요하면 공유 부분을 **하위 레이어로 내려** 거기서 의존한다. 예: 두 모듈이 같은 모델을 쓰면 그 모델을 더 안정적인 하위 레이어에 두고 각자 그것을 의존한다.

### 3. 사이클 금지

- 어떤 경로로도 의존이 자기 자신으로 돌아오지 않는다. A → B → A, A → B → C → A 모두 금지.

## 경계 책임 규칙

레이어 방향 위에 책임 경계가 겹친다. 정확한 경계는 프로젝트 `architecture.md`가 정의하고, 아키텍처에 따라 다음 같은 형태로 나타난다 (구체 어휘·타입명은 프로젝트를 따름).

- **추상은 안정적인 레이어, 구현은 그것을 쓰는 레이어.** 인터페이스(예: Repository protocol)는 하위 레이어가 소유하고, 구현은 인프라를 아는 레이어가 가진다. 상위는 인터페이스에만 의존한다.
- **전송·직렬화 타입은 경계 안에 가둔다.** 인프라 표현(예: DTO)을 도메인 레이어로 노출하지 않는다. 경계에서 도메인 타입으로 변환해 넘긴다.
- **도메인 타입은 전송 포맷에 묶이지 않는다.** 도메인 엔티티가 직렬화 프로토콜을 직접 준수하면 전송 포맷 변경이 도메인을 흔든다. 직렬화 형태는 전송 타입이 진다.
- **인프라 오류는 경계에서 번역한다.** 인프라 오류(네트워크·디코딩 등)를 도메인 오류로 옮겨 던지고, 상위 레이어는 도메인 오류만 다룬다.

## 경계 판정 — 모듈을 어디에 둘까

새 모듈·분리 대상을 배치할 때:

1. **하는 일 한 줄**을 적는다.
2. 그 일이 프로젝트 레이어 정의(`architecture.md`)의 어느 책임에 해당하는지 대조한다 — 화면·사용자 흐름인가 / 데이터 접근·인프라 연동인가 / 화면 없이 도메인 규칙만인가 / 전송 추상인가 / 공용 UI인가 / 레이어 무관 기반인가.
3. 둘 이상에 걸치면 **책임을 쪼개** 각 조각을 맞는 레이어로 나눈다. 한 모듈이 여러 레이어 책임을 동시에 지지 않게 한다.
4. 애매하면 **더 하위(더 의존받는) 레이어로 내리는 쪽**을 우선 검토한다 — 상위는 교체가 싸고, 하위로 잘못 내린 것은 의존자가 생긴 뒤 되돌리기 비싸다.

## 흔한 위반과 교정

| 위반 | 증상 | 교정 |
|---|---|---|
| 역방향 | 하위 레이어가 상위 레이어를 import | 끌어쓰던 타입을 더 하위 레이어로 내리고 의존 방향 뒤집기 |
| 수평 | 동일 레이어 모듈끼리 서로 import | 공유 부분을 하위 레이어로 추출해 양쪽이 그것을 의존 |
| 전송 타입 누출 | 도메인·상위 레이어가 인프라 전송 타입(DTO 등)을 직접 다룸 | 전송 타입을 경계 안에 가두고 도메인 타입으로 변환 |
| 오류 누출 | 상위 레이어가 인프라 오류를 직접 catch | 경계에서 도메인 오류로 번역해 던지기 |
| 기반 레이어 비대 | 최하위 공용 레이어가 특정 상위 전용 로직을 가짐 | 그 로직을 해당 상위 레이어로 올림 |
| 우회 사이클 | A→B→C→A 형태 | 사이클을 끊는 추상(인터페이스)을 하위 레이어에 두고 의존 역전 |

## 빌드가 못 잡는다는 점

위 방향 원칙은 **모두 `dependencies`에 선언하면 컴파일러를 통과한다.** 역방향·수평 의존을 `Package.swift`의 `dependencies`에 적으면 빌드는 초록불이 된다. (미선언 모듈 import는 SPM이 컴파일 실패로 잡지만, 선언된 역방향은 못 잡는다.) 그래서 방향 준수는 빌드가 아니라 [deps-verify.md](deps-verify.md)의 track B로 검증한다 — `Package.swift`의 `dependencies` 배열을 추출해 프로젝트 `layer-rules.json`과 대조하는 정적 검사기 `check_spm_layers.py`(`--rules`에 프로젝트 `layer-rules.json` 지정)가 역방향·수평·사이클을 기계적으로 잡는다.
